# Leetcode 刷题助手项目面试指南 (校招/实习版)

## 0. 项目部署与分享

### 如何分享给别人使用？
只要你有公网 IP，别人就可以通过浏览器访问你的服务器。
URL 格式：`http://你的公网IP:8000/leetcode`

**注意点**：
1.  **防火墙/安全组**：如果你用的是云服务器（阿里云/腾讯云），记得去控制台的安全组规则里**开放 8000 端口**（TCP协议）。
2.  **后台运行**：你的服务器程序必须一直跑着，别人才能访问。如果不小心关了终端程序就会停。

**推荐命令（让程序在后台一直跑）**：
```bash
# 假设你在 build 目录下
nohup ./http_server -p 8000 > server.log 2>&1 &
```
*   `nohup ... &`：让程序在后台运行，即使你断开 SSH 连接也不会停。
*   `> server.log`：把输出日志存到这个文件里，方便排查问题。（`2>&1` 是把错误日志也存进去）

## 1. 核心面试题 (必问)

### Q1: 你的项目中 MySQL 是怎么存放题目的？这样做的好处和坏处是什么？

**回答**：
在这个项目中，我设计了一张 `study_records` 表来存放用户的做题记录。
主要字段包括：`user_id`（用户ID）、`problem_id`（题目ID）、`description`（题目描述）、`test_cases`（测试用例）、`status`（掌握状态）和 `next_review_time`（下次复习时间）。

*   **好处**：
    1.  **查询方便**：利用 MySQL 的索引（user_id + problem_id），可以毫秒级查出某个用户对某道题的复习进度。
    2.  **持久化且可靠**：MySQL 是关系型数据库，保证数据不会丢失，且支持事务（虽然简单版本没重度依赖事务，但为未来扩展留了这种可能）。
    3.  **结构化**：方便进行统计（比如统计多少题已掌握，多少题待复习）。

*   **坏处**：
    1.  **文本占用大**：题目描述 (`description`) 是个大段文本，存放在关系型数据库里会占用较多存储空间，且可能影响查询性能（如果没有做专门优化）。
    2.  **Schema 变更麻烦**：如果以后要加字段（比如“题目标签”），修改大表结构如果是生产环境会比较麻烦（锁表）。
    *   *(补充亮点)*：所以我实现了简单的**自动迁移 (Auto-Migration)** 功能，如果检测到缺少列，程序启动时会自动执行 `ALTER TABLE` 添加列，避免每次部署都要手动改表。

### Q2: RabbitMQ 在这个项目中起了什么作用？为什么要用到它？

**回答**：
**RabbitMQ 主要用于“AI 辅助功能”的日志记录。**
虽然刷题记录（做的题、复习进度）是直接同步写入 MySQL 的，但**用户与 AI 的每一次对话（包括代码分析、判题请求）**，我是通过 RabbitMQ 异步存储到 `chat_message` 表中的。

*   **为什么要用（两大作用）**：
    1.  **削峰填谷 (Traffic Shaping)**：如果以后扩展成“万人并发判题”，大量的 AI 对话日志写入会瞬间击垮数据库。用 MQ 缓冲一下，后台慢慢写库，保护数据库。
    2.  **解耦 (Decoupling)**：AI 判题的过程可能很慢（因为要等大模型返回），我不希望“保存对话记录”这个动作阻塞主线程，或者因为数据库写入抖动而影响用户拿到 AI 结果的速度。
    *   *(面试技巧)*：面试官如果问“为什么刷题记录不用 MQ？”，你可以说：“因为刷题记录数据量相对小（用户一天做不了几道题），且用户希望看到‘即时保存成功’的反馈，所以选择了同步写库。而 AI 对话日志数据量大且实时性要求不高，适合异步。”

### Q3: 简要介绍一下 HttpServer 是怎么搭建起来的？包含哪些模块？

**回答**：
我的 HttpServer 是基于 **Muduo 网络库**（Reactor 模型 + 非阻塞 I/O）搭建的高性能服务器。
主要由以下几个模块组成：

1.  **TcpServer (核心网络层)**：直接复用 Muduo 的 TcpServer，负责处理底层的 TCP 连接、数据收发。之所以用它，是因为它采用 **Epoll + 线程池** 模型，能处理高并发连接，比传统的“一线程一连接”效率高得多。
2.  **HttpContext (协议解析层)**：负责解析 HTTP 协议。它是一个状态机，把 TCP 收到的字节流解析成 `HttpRequest` 对象（包含 Header、Body、Method 等）。这样做是为了**解耦网络通信与业务逻辑**。
3.  **Router/Dispatcher (路由分发层)**：根据请求的 URL（比如 `/leetcode/record`），找到对应的 **Handler**（处理器）。这样写代码条理清晰，一个 URL 对应一个类，易于维护。
4.  **DbConnectionPool (数据库连接池)**：预先创建好一组 MySQL 连接。因为建立 TCP 连接很耗时，复用连接池能显著提高数据库操作的响应速度。

### Q4: 你的项目中是如何实现“艾宾浩斯复习曲线”的？

**回答**：
核心逻辑在 `LeetcodeHandler` 中。
我在数据库里存了一个 `stage`（当前阶段）和 `next_review_time`（下次复习时间戳）。
具体的算法是：
*   当用户完成复习时，`stage + 1`。
*   根据新的阶段，计算间隔时间：1天、2天、4天、7天、15天...
*   `next_review_time = 当前时间 + 间隔时间`。
*   每次用户打开首页，我只需要查 `WHERE next_review_time <= 当前时间`，就能把所有该复习的题目拉出来。

### Q5: 为什么要把前端代码 (HTML) 放在后端项目里？通常不都是前后端分离吗？

**回答**：
（这是个好问题，也是个坑）
在商业项目中确实是前后端分离（Vue/React + Nginx + API Server）。
但在本项目（校园/个人项目）中，为了**部署简单**和**演示方便**，我选择由 C++ 服务器直接提供静态资源服务。
这样做的好处是**开箱即用**：只需要编译一个二进制文件，不需要额外部署 Nginx，就能跑起来完整的应用。虽然耦合度略高，但对于单体应用来说，维护成本最低。

### Q6: AI 判题功能是怎么实现的？

**回答**：
本质上是 **Prompt Engineering (提示词工程)**。
后端并没有真的去编译运行用户的 C++ 代码（那样做太危险，容易被注入攻击），而是把题目描述和用户代码打包成一段提示词，发给大模型（如通义千问/Deepseek）。
提示词里明确要求：“请作为判题机，判断代码逻辑是否正确，只需返回 JSON 格式的 Pass/Fail...”。
大模型的代码理解能力非常强，对于算法题的逻辑判断准确率很高，这是一种利用 AI 能力快速实现复杂功能的典型应用。
